Discuss the scope of var, let, and const.


var

Function Scope: A variable declared with var is function-scoped. This means that the variable is accessible anywhere within the function it was declared in, even if it's declared inside a block like a loop or an if statement.

Global Scope: If declared outside any function, it has a global scope and is accessible from anywhere in your code, potentially leading to unintended side-effects.

Hoisting: Variables declared with var are hoisted to the top of their scope. This means they are technically available even before the line on which they are declared. However, their value will be undefined until the line where you actually initialize them.

let

Block Scope: let was introduced in ES6/ES2015 and provides block scoping. A variable declared with let inside a block (like inside an if statement, a loop, or any curly braces {}) is only accessible within that block.

Global Scope: If declared outside any function or block, it's global but not added to the global object (like window in browsers).

Temporal Dead Zone:While let variables are also hoisted, they have a "temporal dead zone" from the start of the block until the declaration is encountered. Any reference to the variable in this "dead zone" causes a ReferenceError.

const

Block Scope: Just like let, const is block-scoped.

Immutability:When you declare a variable with const, its value can't be reassigned. This doesn’t mean the value is immutable, just that the variable identifier cannot be reassigned. For instance, if you have a const object, you can still modify its properties.

Temporal Dead Zone:It also has the same hoisting behavior with the "temporal dead zone" as let.


It's important to note that nowadays, using let and const is generally preferred over var due to their block scoping, which can help avoid various bugs and make code easier to understand. It's recommended to use const by default and only use let when you know the value will change. Avoid var unless you have a specific reason or are working on older codebases.

Tell us the use cases of null and undefined.


undefined

Variables:When a variable is declared but not initialized, it's given the value of undefined.

Functional Parameters:When a function expects a parameter, but it's not provided, the parameter takes the value of undefined.

Object Properties:When you try to access a property of an object that doesn't exist, you get undefined.

Return Values:If a function doesn't return anything, it returns undefined by default.

Array Elements: IAccessing an array index that doesn't exist will yield undefined.

null

Intentional Absence:null is generally used to represent a deliberate non-value or absence of any object value. It's a way for a developer to signal that there should be "nothing" in a variable, property, or slot.

Object Properties: When you want to express that an object's property exists but currently holds no value (or you want to reset its value), you can assign it null.

Prototypal Ancestor:The prototype chain ends with null. When you access the __proto__ property of Object.prototype, the value is null.

DOM: When querying the DOM for elements, if no elements are found, methods like document.getElementById() return null.


To make your code more expressive, prefer to use null when you want to intentionally denote that something has "no value", and let undefined indicate something was unintentionally left uninitialized or not provided.

What do you mean by REST API?


A REST API (also known as RESTful API) is an application programming interface (API or web API) that conforms to the constraints of REST architectural style and allows for interaction with RESTful web services. REST stands for representational state transfer and was created by computer scientist Roy Fielding.

REST is a set of architectural constraints, not a protocol or a standard. API developers can implement REST in a variety of ways.
When a client request is made via a RESTful API, it transfers a representation of the state of the resource to the requester or endpoint. This information, or representation, is delivered in one of several formats via HTTP: JSON (Javascript Object Notation), HTML, XLT, Python, PHP, or plain text. JSON is the most generally popular file format to use because, despite its name, it’s language-agnostic, as well as readable by both humans and machines.
Something else to keep in mind: Headers and parameters are also important in the HTTP methods of a RESTful API HTTP request, as they contain important identifier information as to the request's metadata, authorization, uniform resource identifier (URI), caching, cookies, and more. There are request headers and response headers, each with their own HTTP connection information and status codes.


In order for an API to be considered RESTful, it has to conform to these criteria:

  1. A client-server architecture made up of clients, servers, and resources, with requests managed through HTTP.
  2. Stateless client-server communication, meaning no client information is stored between get requests and each request is separate and unconnected.
  3. Cacheable data that streamlines client-server interactions.
  4. A layered system that organizes each type of server (those responsible for security, load-balancing, etc.) involved the retrieval of requested information into hierarchies, invisible to the client.
  5. Code-on-demand (optional): the ability to send executable code from the server to the client when requested, extending client functionality.

In contrast, REST is a set of guidelines that can be implemented as needed, making REST APIs faster and more lightweight, with increased scalablity—perfect for Internet of Things (IoT) and mobile app development.
